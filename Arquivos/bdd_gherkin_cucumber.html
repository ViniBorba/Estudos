BDD - Behaviour Driven Development (Desenvolvimento Orientado a Comportamento)

É uma maneira de trabalho que envolve no mesmo círculo pessoas do negócio e pessoas técnicas no âmbito do desenvolvimento.

Também é uma maneira de trabalho que testa e valida o comportamento do sistema.
Por consequência disso acaba gerando documentação do projeto.

No BDD é incentivado colaboração entre 3 áreas distintas: área de negócios, desenvolvedores e testes\qualidade.

BDD aborda partes pequenas do desenvolvimento, o que acaba agregando rápido entendimento do que está sendo desenvolvido.
Chamamos essas pequenas partes de cenários ou exemplos concretos.

O BDD tem algumas semelhanças com abordagens ágeis.
Mas o BDD não substituí essas abordagens, ele complementa.

Essencialmente a abordagem do BDD pode ser descrita em 3 etapas:

  • Discovery (Descoberta)
  • Formulation (Formulação)
  • Automation (Automação)

Cada uma dessas etapas tem um papel específico:

Discovery - o que poderia fazer

> É a primeira etapa, e a mais importante, é onde acontecem as definições para as etapas seguintes.

> A comunicação é parte fundamental para desenvolver um software valioso e funcional.
Conversas entre as pessoas que imaginaram o sistema e as pessoas que entregam esse sistema pode fazer muita dierença no final.

> O BDD ajuda as equipes a ter as conversas certas nos momentos certos, para minimizar o tempo gasto em reuniões e maximizar a quantidade de código valioso.
Essas conversas são chamdas de workshops de descoberta. O Gherkin e o Cucumber vão estruturar essas conversas.

> Workshops de descoberta é uma reunião de no máximo 30 minutos entres os três amigos:

  • a pessoa de negócios
  • o desenvolvedor
  • pessoa dos testes e qualidade

> É nos Workshops de descoberta que são construídos oscenários, também chamado deexemplos concretos.

> Os exemplos concretos são construídos tendo como base ações concretas, sempre levando em conta a perspectiva dos usuários.
Podem ser utilizados os User Stories para iniciar a construção dosexemplos concretos.

> Essas conversas aumentam o entendimento compartilhado entre as equipes.
> Isso vai trazer um melhor entendimento do projeto para todos, gerando novas perspectivas, descobrindo lacunas, e priorizando entregas que geram maior valor.

Formulation - o que deve fazer

> No Workshop de descoberta, assim que é identificado um exemplo concreto, criamos uma documentação para ele.

> Essa documentação é gerada de forma estruturada que pode ser lido tanto por humanos quanto por computadores.

> A linguagem para escrever essa estrutura é oGherkin.
> É gerado um arquivo .feature que o Cucumber é capaz de interpretar.

> Como essa linguagem é compreendida por todos, podemos obter feedback de todas as equipes sobre visão construída.
> Essa colaboração integrada traz uma linguagem compartilhada e única para falar sobre o sistema.

Automation - o que ela realmente faz

> Agora que temos nossa especificação podemos usá-la para guiar nossa implementação no desenvolvimento.

> Uma vez que temos um exemplo concreto escrito e executável no arquivo .feature, nós testamos esse exemplo no nosso projeto através do Cucumber.

> O teste vai falhar porque ainda não implementamos o comportamento que ele descreve.

> Junto a mensagem de falha o Cucumber traz parte da implementação para alcançar o comportamento descrito.

Exemplo: 

            
                Step undefined
                You can implement this step and 2 other step(s) using the snippet(s) below:

                @Given("Lucy está localizada a {int} metros de Sean")
                public void lucy_está_localizada_a_metros_de_sean(Integer int1) {
                    // Write code here that turns the phrase above into concrete actions
                    throw new io.cucumber.java.PendingException();
                }
                
                @When("Sean grita {string}")
                public void sean_grita(String string) {
                    // Write code here that turns the phrase above into concrete actions
                    throw new io.cucumber.java.PendingException();
                }
                
                @Then("Lucy ouve a mensagem de Sean")
                public void lucy_ouve_a_mensagem_de_sean() {
                    // Write code here that turns the phrase above into concrete actions
                    throw new io.cucumber.java.PendingException();
                }

> Agora é desenvolvido o código de implementação usando como base o retorno da falha no primeiro teste, ele vai servir como um guia conforme necessário.

Gherkin e Cucumber

O Cucumber lê especificações executáveis escrita em texto simples e valida se o software faz o que essas especificações dizem.

Essas especificações são construídas tendo como base os cenários, também chamados de exemplos concretos, criados seguindo as premissas do BDD.

Para que o Cucumber entenda as especificações criadas, elas devem seguir algumas regras básicas de sintaxe.

Exemplo de código: 

        
            Feature: Ouvir grito
            
                Scenario: O ouvinte está dentro do alcance
                Given Lucy está localizada a 15 metros de Sean
                When Sean grita "café de graça"
                Then Lucy ouve a mensagem de Sean

Cada exemplo concreto torna-se uma especificação que o Cucumber verifica e valida, gerando relatórios com sucessos e falhas.

Gherkin é um conjunto de regras gramaticais que torna o texto estruturado simples o suficiente para o Cucumber entender.
Gherkin é o "idioma" do Cucumber.

Os documentos do Gherkin são armazenados em arquivos .feature e normalmente são versionados no controle de origem junto com o software.

Gherkin, Cucumber e BDD trabalham juntos para trazer: documentação, testes automatizados e expecificações.

Uso em um projeto

De modo geral, a aplicação das ferramentas do BDD devem sempre ser o mais tardio possível antes do início do desenvolvimento.
